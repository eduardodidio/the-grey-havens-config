## User Story

As a security-conscious maintainer, I want the drivers' elevated-permission
posture documented and guarded, so that the `--dangerously-skip-permissions` /
`--yolo` flags are an explicit, audited decision and can be neutralized for
dry-run/testing without changing default behavior.

## Objective

Add a **Security** section to `drivers/DRIVER_CONTRACT.md` describing the
threat model and elevated permissions, add an opt-in safe/dry-run hook to the
drivers that prints the command instead of executing (default behavior
unchanged — F01 AC3), and add a guardrail test.

## Implementation details

- `drivers/DRIVER_CONTRACT.md`: new "## Security posture" section —
  drivers run the provider CLI with **elevated/auto-approved permissions**
  (autonomous agents); list the exact flags per driver; state the mitigations
  (isolated bash context per spawn, JSONL audit log, CLAUDE.md Highlander/
  no-secrets sandbox rule, `.gitignore` excludes transcripts); document
  `DIDIO_DRIVER_DRYRUN`.
- `drivers/claude-driver.sh` + `codex-driver.sh`: wrap the real invocation so
  that when `DIDIO_DRIVER_DRYRUN` is set (non-empty), the driver prints the
  resolved command (to the log file, as one NDJSON line) and exits 0 **without
  calling the CLI**; otherwise behaves exactly as today.
- `tests/F02-driver-guardrails.sh`: assert (a) the dangerous flags are still
  present in each driver (so silent removal is caught), (b) each flag is
  mentioned in `DRIVER_CONTRACT.md` (doc/code parity), (c) `DIDIO_DRIVER_DRYRUN`
  short-circuits without invoking `claude`/`codex` (use a fake bin on PATH or
  assert the dry-run NDJSON line + no error).

## Acceptance criteria

- [ ] `DRIVER_CONTRACT.md` has a Security section enumerating each driver's
      elevated flags and the mitigations.
- [ ] Default driver behavior unchanged (no new flag in the real invocation;
      `DIDIO_DRIVER_DRYRUN` unset → identical command). F01 driver tests green.
- [ ] `DIDIO_DRIVER_DRYRUN=1` makes claude/codex drivers print-and-exit-0
      without invoking the provider CLI.
- [ ] Guardrail test fails if a dangerous flag is removed from a driver but
      not from the doc, or vice-versa (parity).

## Testing

- Framework: bash. New `tests/F02-driver-guardrails.sh` (via `tests/run.sh`).
- For dry-run, set `DIDIO_LOG_FILE` to a temp file and assert the printed
  command line; ensure no real `claude`/`codex` call (PATH shim or absence).

## Test scenarios

- **Happy path (dry-run):** `DIDIO_DRIVER_DRYRUN=1` → resolved command logged,
  exit 0, provider CLI not called.
- **Edge case:** doc/code parity — every dangerous flag in code appears in
  the contract doc.
- **Error scenario:** a driver missing its expected dangerous flag → test
  fails (guards against accidental weakening or silent escalation).
- **Boundary:** `echo-driver` has no elevated flags and is exempt.
