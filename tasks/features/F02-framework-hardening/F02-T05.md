## User Story

As a dashboard consumer, I want the initial `.meta.json` built safely, so that
a task id / file path containing a quote or newline can't produce invalid JSON
or inject content.

## Objective

Replace the unquoted heredoc that writes the initial `.meta.json` in
`bin/didio-spawn-agent.sh` (lines ~75–89) with a `python3`/`json.dump` build
using `sys.argv`/env, mirroring the already-correct final rewrite at
lines ~158–171. Observable JSON for clean inputs stays equivalent.

## Dev Notes

Veja `_brief/00-overview.md` e `_brief/04-security-hardening.md` (D3).

- **Vulnerable block** — `bin/didio-spawn-agent.sh:75-89`:
  ```bash
  cat > "$META_FILE" <<EOF
  { "feature": "$FEATURE", "role": "$ROLE", "task": "$TASK_ID",
    "task_file": "$TASK_FILE", ... "pid": $$, ... }
  EOF
  ```
  An unquoted `<<EOF` interpolates `$FEATURE/$ROLE/$TASK_ID/$TASK_FILE`
  straight into JSON — a `"` or newline breaks parsing.
- **Correct pattern already present** at lines 158–171:
  `python3 - "$META_FILE" ... <<'PY' ... json.dump ... PY`.
- Preserve all current fields and values: `feature, role, task, task_file,
  started_at` (`date -u +%Y-%m-%dT%H:%M:%SZ`), `log, status:"running", pid`
  (`$$`), `model` (`${AGENT_MODEL:-default}`), `fallback_model`
  (`${AGENT_FALLBACK:-none}`), `provider`. Keep `indent` consistent with the
  final rewrite so the two writes don't fight (`json.dump(..., indent=2)`).
- Do **not** touch the resolution logic, the driver dispatch, or the final
  rewrite — only the initial header construction. This is the only Wave-1
  writer of `didio-spawn-agent.sh`.
- Keep `set -euo pipefail` (top-level executable — correct as-is).

## Implementation details

- Build the initial meta by passing the fields as argv to
  `python3 - "$FEATURE" "$ROLE" "$TASK_ID" "$TASK_FILE" "$LOG_FILE" "$$"
  "${AGENT_MODEL:-default}" "${AGENT_FALLBACK:-none}" "$PROVIDER" <<'PY'`,
  computing `started_at` inside Python (`datetime.now(timezone.utc)`) or
  passing it in. Write with `json.dump(..., indent=2)` + trailing newline.
- Add `tests/F02-spawn-meta.sh` asserting valid JSON for hostile inputs.

## Acceptance criteria

- [ ] Initial `.meta.json` is produced via `json.dump` from argv/env — no
      shell variable interpolated into a JSON heredoc.
- [ ] A task file path / id containing `"` or a newline yields **valid,
      parseable** JSON (`python3 -m json.tool` succeeds) with the literal value.
- [ ] For clean inputs, the meta JSON has the same fields/values as before
      (diff only in whitespace/formatting, not content).
- [ ] `bash tests/F01-spawn-dispatch.sh` still green (status running→completed/
      failed unchanged).

## Test scenarios

- **Happy path:** normal spawn → meta has all expected fields, `status`
  transitions running→completed, valid JSON at both write points.
- **Edge case:** task id / path containing `"`/newline → still valid JSON.
- **Error scenario:** driver exits 1 → final meta `status:"failed"`,
  `exit_code:1`; initial header already valid.
- **Boundary:** empty model/fallback default to `default`/`none`; provider
  string preserved.
