## Wave 1 — Remediations (T04–T07, T10, T11)

### T04 — `bin/didio-config-lib.sh` (injection fix D1/D2)

**Architecture:** All `python3 -c "..."` patterns replaced with
`python3 - "$arg1" "$arg2" <<'PY'...\nPY`. Single-quoted heredoc delimiter
(`<<'PY'`) prevents shell expansion inside the body. Data arrives via
`sys.argv[1]`, `sys.argv[2]`, etc. ✓

**Security:** Closes D1/D2 (HIGH). Adversarial inputs (single quote, double
quote, newline, `$(whoami)`, backtick, Python code fragment) are tested by
`F02-config-injection.sh` and all pass. The observable output is preserved:
bool → `true`/`false`, int → int, dict/list → JSON, scalar → str. ✓

**Code quality:** No dead code. `didio_find_feature_dir` still uses `find`
(not a Python call), which is fine — no injection vector there since it only
uses shell variables, not config-read values.

**Tests:** `F02-config-injection.sh` passes. Coverage: happy path, single/double
quote, newline, command substitution, backtick, Python fragment, key with special
chars, very long value, unicode, bool/int typing, role getter functions, error
scenarios (malformed JSON, missing config), economy mode, turbo override,
provider_bin fallback, second_brain defaults. Comprehensive. ✓

### T05 — `bin/didio-spawn-agent.sh` (meta-header hardening D3)

**Architecture:** Both the initial `.meta.json` write and the final meta-rewrite
use `python3 - "$META_FILE" ... <<'PY'...\nPY` with `json.dump`. Values arrive
via `sys.argv`. The 10-argument initial write (`path, feature, role, task,
task_file, log, pid, model, fallback, provider`) is clean. ✓

**Security:** Closes D3 (MEDIUM). A task file path containing `"` or `\n` no
longer breaks JSON. Tests in `F02-spawn-meta.sh` validate both hostile inputs
(Scenario 2: `it"s-hostile.md`, Scenario 3: newline in path).

**Code quality:** `FULL_PROMPT` is assembled via `cat <<PROMPT...PROMPT`
(unquoted heredoc), which is intentional — the `$ROLE_PROMPT`, `$TASK_BODY`,
`$EXTRA` and `$FEATURE`/`$TASK_ID` are expanded into the prompt body. This is
the correct trade-off: the prompt is text content for the LLM, not data
processed by a parser. ✓

**Tests:** `F02-spawn-meta.sh` passes. 5 scenarios: happy path, hostile quote,
hostile newline, failure path, boundary defaults. ✓

2. **`test_didio_events.py`** (1 failure: `test_codex_event_with_no_analogue_degrades_to_raw`):
   Codex reasoning category assertion fails. Pre-existing and out of F02 scope.

### Echo-driver format divergence (TL-03) — IMPORTANT

F02 introduces `drivers/echo-driver.sh` (the real driver for simulation tests)
and `tests/fixtures/F02-probe-driver.sh`. The new echo-driver emits:
```json
{"type":"system","subtype":"echo-driver","role":"...","feature":"...","task":"...","model":"...","fallback":"...","effort":"..."}
```
F01 test `F01-spawn-dispatch.sh` expects:
```
"type":"echo"   (not present)
"role":"developer","feature":"F01","task_id":"task"  (uses "task_id", not "task")
```
The divergence is real: the F01 echo-driver used a different schema. Since
both drivers coexist (F01 used a different echo-driver fixture not checked in,
F02 added `drivers/echo-driver.sh`), the old F01 test assertions are now stale.
A follow-up task should either update `F01-spawn-dispatch.sh` to match the new
canonical echo-driver format or version the echo-driver schemas.

## Retrospective Seeds

- **Pattern:** ADR env-var names diverge from implementation during development —
  the ADR is written early, the implementation settles on a different name, and
  the ADR is never updated.
- **Role(s) affected:** developer, techlead
- **Lesson:** During code review, explicitly cross-check every named constant
  (env var, flag, function name) in ADRs and diagrams against the actual
  implementation. A grep is faster than a re-read.

---

- **Pattern:** Shellcheck target globs only cover top-level test directories,
  missing `tests/lib/` or any sub-directory with shell helpers.
- **Role(s) affected:** developer, architect
- **Lesson:** When adding a `tests/lib/` or similar helper directory, include a
  task or subtask to widen the shellcheck glob pattern. The Architect should
  add this as a Wave 0 acceptance criterion whenever a lib/ sub-directory is
  introduced.

---

- **Pattern:** F01 test assertions become stale when F02 changes shared fixtures
  (echo-driver NDJSON schema) — the cross-feature impact is invisible at task
  level but surfaces in `tests/run.sh` as pre-existing failures that are hard to
  distinguish from real regressions.
- **Role(s) affected:** architect, developer, techlead, qa
- **Lesson:** When a task introduces a new or modified shared driver/fixture,
  explicitly add "check F01 tests still pass with new fixture" as an acceptance
  criterion, not just "F01 tests still green."
