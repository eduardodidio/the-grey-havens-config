### AC3 fidelity decision — RATIFIED

The brief originally called for `claude-driver.sh` to drop `--effort` and
`--allowedTools`. The implementation as committed **keeps both flags**,
because the *current* (pre-F01) `didio-spawn-agent.sh` already passes both —
dropping them would have been a silent behavior regression disguised as a
"plan-faithful" port. I compared `drivers/claude-driver.sh` against the
pre-F01 inline invocation (via `git diff bin/didio-spawn-agent.sh`): the flag
set, order, and `--dangerously-skip-permissions` / `--allowedTools "Edit
Write MultiEdit Read Bash Glob Grep"` string are reproduced byte-for-byte.
**This is the correct call** — AC3 says "byte-for-byte unchanged," and the
plan was written against a stale snapshot. The driver file's inline comment
documents this rationale, which is good practice for future readers. ✅
Ratified, no action needed.

## Wave 2 — F01-T07/T08/T09 (spawn-agent dispatch, codex-driver, compile→Claude)

- `bin/didio-spawn-agent.sh` (`git diff` reviewed in full):
  - Resolves `PROVIDER` via `didio_provider_for_role`, defaulting to
    `"claude"`.
  - Resolves `DRIVER` with **project-local override precedence**
    (`$PROJECT_ROOT/drivers/<provider>-driver.sh` wins over
    `$DIDIO_HOME/drivers/<provider>-driver.sh`) — consistent with the
    project-local-first precedence convention already used for
    `didio.config.json` / `didio-config-lib.sh`.
  - **Ordering is correct and well-commented**: the driver-existence check
    (`[[ ! -x "$DRIVER" ]]` → `exit 2`, "unknown/unsupported provider") runs
    **before** the `didio-providers.sh validate-roles` preflight (`exit 1`,
    "missing CLI"). This gives two distinct, debuggable failure modes:
    unsupported provider vs. supported-but-uninstalled CLI. The
    reconciliation note in the task description said this ordering was the
    key fix from the project-local→DIDIO_HOME port; confirmed it landed
    correctly. ✅
  - `.meta.json` gains a `"provider"` field — good for dashboard/observability
    (Wave 3 consumes this).
  - Driver contract env vars (`DIDIO_PROMPT`, `DIDIO_MODEL`, `DIDIO_FALLBACK`,
    `DIDIO_EFFORT`, `DIDIO_TASK_ID`, plus pre-existing `DIDIO_ROLE`,
    `DIDIO_FEATURE`, `DIDIO_LOG_FILE`) are exported before dispatch.
  - `DIDIO_DRY_RUN=1` path branches on provider: Claude prints the exact flag
    preview (unchanged from before), Codex prints a reduced
    model/fallback/effort preview. Reasonable — Codex's CLI shape differs
    enough that a byte-identical preview isn't meaningful.
  - Final dispatch is `"$DRIVER"` — clean, single point of provider variance.
- `drivers/codex-driver.sh`: maps to `codex exec --json --yolo
  ${--model}`, prompt via stdin (documented rationale: avoids arg-length /
  multiline issues). Header comments explicitly document **known gaps**:
  `DIDIO_FALLBACK` and `DIDIO_EFFORT` are accepted but unused (Codex has no
  equivalent) — this satisfies AC4's "or degrade gracefully with a documented
  gap" requirement. ✅
- `bin/didio-compile-skills.sh` / `bin/didio-compile-skills.py`: compiles
  `skills/*.md` → `.claude/commands/`, `.claude/agents/`,
  `agents/prompts/`. `tests/F01-compile-claude.sh` passes.
- **Architecture**: dispatch is centralized in one script, drivers are
  swappable, no role-specific logic leaked into `didio-spawn-agent.sh`. ✅

## Retrospective Seeds

- **Pattern:** When a feature changes a shared generated-artifact pipeline
  (here: `skills/` → `didio compile-skills` → `templates/.claude/commands/` →
  synced into `.claude/commands/`), the *target* repo's already-synced copies
  drift out of byte-identity with the new template output, breaking
  byte-identity tests (F10) even though the generator itself is correct.
  **Role(s) affected:** developer, qa. **Lesson:** after wiring a new
  compile/sync step, run it once against the shipping repo (or note it as a
  required one-time migration step) so byte-identity tests reflect
  post-migration state, not pre-migration state.

- **Pattern:** A backward-compatible *additive* config schema change (new
  top-level key in `templates/didio.config.json`) is correct per AC3/AC1, but
  any existing-state fixture asserting `NO_CHANGE`/idempotency on that config
  file (F12 scenario 2) will need its baseline fixture updated in the same
  PR. **Role(s) affected:** architect, developer. **Lesson:** when a task
  touches `templates/didio.config.json`'s schema, grep for tests asserting
  config-sync idempotency (`NO_CHANGE`, "already up to date") and update their
  fixtures in the same task, not as an afterthought.
