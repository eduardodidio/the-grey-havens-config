# Task F01-T13 — Preflight: validate provider CLIs before a Wave

**Feature:** F01
**Wave:** 3
**Type:** backend (CLI / gate)
**Depends on:** F01-T04, F01-T08
**Status:** done

## User Story

As an operator, I want a Wave to abort early if a configured provider's CLI is
missing or unauthed, so that I don't waste a half-run discovering it mid-Wave.

## Objective

Add a preflight check to `bin/didio-run-wave.sh` that validates every provider
used by the Wave's roles before launching any agent.

## Dev Notes

Veja `_brief/00-overview.md` e `_brief/03-cli-config-surface.md` (section
"Preflight"). Reuses `didio providers validate` (T12) and
`didio_provider_for_role` (T04).

- File: `${DIDIO_HOME:-~/.claude-didio-config}/bin/didio-run-wave.sh`. It parses
  the Wave manifest from `tasks/features/<feature>/<feature>-README.md` and spawns
  one `didio-spawn-agent` per task.
- Before spawning: collect the distinct providers across the roles that will run
  this Wave (or simply call `didio-providers.sh validate` for the active config).
  On non-zero → print which binary is missing + an install/auth hint, then abort
  the Wave (exit non-zero) WITHOUT spawning.
- **Backward-compat:** a Claude-only project where `codex` isn't installed and no
  role uses it must NOT be blocked. Only validate providers actually in use.
- Optional: also run preflight in `didio-spawn-agent.sh` for the single-agent
  path (lighter check: just the one role's provider). Keep it cheap.

## Implementation details

- Prefer delegating to `bin/didio-providers.sh validate` to avoid duplicating
  `command -v` logic. Capture its exit code; surface its message.
- Place the check after manifest parse, before the spawn loop. Do not change the
  parallelism/spawn logic itself.

## Testing

- Framework: bash integration test with manipulated PATH.
- Command: `bash ${DIDIO_HOME}/tests/F01-preflight.sh`
- Test file: `${DIDIO_HOME}/tests/F01-preflight.sh`
- Convention: temp feature dir with a README manifest + config; toggle a provider
  binary present/absent on PATH; assert abort vs proceed. Stub spawn-agent so no
  real model runs.

## Test scenarios

- **Happy path:** all used providers present → preflight passes, Wave proceeds.
- **Edge case:** Claude-only Wave with `codex` absent → preflight passes (codex
  not used).
- **Error scenario:** a role uses `codex`, `codex` absent → Wave aborts non-zero,
  message names `codex` + hint, no agent spawned.
- **Boundary values:** mixed Wave (one claude role + one codex role), codex
  present, claude absent → aborts naming `claude`.
