## User Story

As a framework maintainer, I want to declare a `provider` per role and register
the available providers in `didio.config.json`, so that the execution layer can
route each role to Claude or Codex without breaking existing Claude-only configs.

## Objective

Add a backward-compatible `providers` registry and an optional per-role
`provider` key to the config schema and the framework template.

## Implementation details

- Add top-level `providers`:
  ```json
  "providers": {
    "claude": { "bin": "claude", "default": true },
    "codex":  { "bin": "codex" }
  }
  ```
- Allow optional `"provider": "claude"|"codex"` inside any `models.<role>` and
  `models_economy.<role>` entry. In the **template**, include a commented example
  (JSON has no comments — use a sibling `_comment` key or a documented example in
  the README/SPEC, not inline `//`). Keep ALL roles on Claude by default.
- Do not wire any code here — schema + template only. Helpers come in T04.

## Test scenarios

- **Happy path:** `python3 -c "import json;json.load(open(cfg))"` succeeds;
  `providers.claude.bin == "claude"`, `providers.codex.bin == "codex"`.
- **Edge case:** a role with no `provider` key still loads; schema additions did
  not alter any existing `models.<role>.model` value (diff-check vs git).
- **Error scenario:** malformed JSON is caught by the load assertion (guards
  against trailing-comma regressions).
- **Boundary values:** template config has zero roles defaulted to `codex`
  (grep proves no `"provider": "codex"` in the shipped template).
