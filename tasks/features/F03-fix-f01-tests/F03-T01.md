# F03-T01 — Root-cause audit & direction confirmation

**Wave:** 0
**Type:** docs (read-only investigation)
**Depends on:** (none)

## User Story

As a maintainer, I want a written, evidence-backed root-cause for each of the
two failing test files, so that the Wave 1 fix tasks change the right artifact
(test vs. production) without guessing and without risking the F02 suites.

## Objective

Investigate and document why the 2 files fail under `tests/run.sh`, and confirm
the fix direction for each (align test ↔ canonical contract; mapper bug). This
task writes **only** a findings document — no code or test edits.

## Dev Notes

Veja `_brief.md` (raiz da feature). Este é um Wave 0 de root-cause; produza
fatos verificáveis, não opiniões.

Project conventions (`CLAUDE.md`): bash stack, tests via `bash tests/run.sh`;
the driver contract is canonical (`drivers/DRIVER_CONTRACT.md`). Guardrails:
não desabilitar/alterar produção "só pra passar teste"; em dúvida, parar e

Key files to inspect (read-only):

- `drivers/echo-driver.sh` (linha 11 schema; linha 15 `exit "${ECHO_DRIVER_EXIT:-0}"`)
- `drivers/DRIVER_CONTRACT.md` (linha 33: var canônica `ECHO_DRIVER_EXIT`)
- `bin/didio-spawn-agent.sh` (linhas 152–160 e 193: captura/propagação de exit)

- `tests/fixtures/codex-json.jsonl` (linha 1: item `reasoning`)

Gotcha (exit-code): `VAR=1 cmd` exporta `VAR` para o ambiente de `cmd` e este é
herdado pelos filhos. Logo `ECHO_DRIVER_EXIT=1 spawn-agent ...` chega ao driver;
`DIDIO_ECHO_EXIT_CODE=1` nunca é lido por ninguém. Prove isso experimentalmente
(comando abaixo) antes de concluir.

## Implementation details

Criar `tasks/features/F03-fix-f01-tests/F03-root-cause.md` contendo, por falha:
- Trecho exato do assert que falha (arquivo:linha).
- Trecho exato do código canônico que define o comportamento correto.
- Conclusão: o fix toca **teste** ou **produção**? Qual arquivo/linha?
- Para 1B explicitamente: veredito de que a propagação de exit em
  `bin/didio-spawn-agent.sh` está correta (com a evidência do experimento) e que
  o defeito é o nome de variável no teste.

Comandos de evidência (rodar e colar a saída no documento):
```bash
# Mostra que o teste usa o nome errado e o driver/contrato usam o certo
grep -n 'DIDIO_ECHO_EXIT_CODE\|ECHO_DRIVER_EXIT' tests/F01-spawn-dispatch.sh drivers/echo-driver.sh drivers/DRIVER_CONTRACT.md

# Prova empírica de que a propagação de produção funciona com o nome CORRETO:
T=$(mktemp -d); mkdir -p "$T/agents/prompts" "$T/logs/agents"
echo '# dev' > "$T/agents/prompts/developer.md"; echo '# task' > "$T/task.md"
cat > "$T/didio.config.json" <<'JSON'
{"providers":{"claude":{"bin":"claude","default":true},"echo":{"bin":"true"}},
 "models":{"developer":{"model":"sonnet","fallback":"haiku","provider":"echo"}}}
JSON
( cd "$T" && DIDIO_HOME="$PWD" ECHO_DRIVER_EXIT=1 \
  "$OLDPWD/bin/didio-spawn-agent.sh" developer F01 task.md >/dev/null 2>&1 ); echo "exit=$?"
# Esperado: exit=1  (prova: produção propaga; o teste é que usava o nome errado)
rm -rf "$T"
```
